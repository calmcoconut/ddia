The Scenario

You are designing a distributed file locker service. When a user opens a file for editing in their web client, the client requests an edit lease from a central locking service. The lock service grants a lock with a 30-second lease. If the user makes edits, the client sends write requests containing the updates directly to a storage server. The locking service and the storage server are independent systems.

Interleaved Scenario Challenge

Mixes all topics · Builds discrimination · Forces transfer · ~10 min

Each scenario requires you to combine knowledge from multiple sections of Chapter 9. This deliberate mixing forces your brain to select the right framework — the hallmark of expert-level understanding.

Scenario 1 of 4

🕒 High-Frequency Algorithmic Trading Platform

You are designing the transaction log for a global high-frequency trading platform. Orders from Tokyo, New York, and London must be matched in the exact order they were received. If two orders arrive within microseconds of each other, the order that was placed first must win. The operations team suggests using NTP-synchronized physical servers to timestamp the order at the local host.

Scenario 2 of 4

🧟 ZooKeeper Leases and the Storage Zombie

A distributed document editor uses ZooKeeper to grant edit leases to writers. Node A gets an edit lease for "budget.docx" with a transaction ID (zxid) of 105. Node A checks if the lease is valid, it returns true, but then Node A's JVM triggers a major 'stop-the-world' garbage collection pause that halts all threads for 20 seconds.

Scenario 3 of 4

⚖️ The Asymmetric Network Partition

In a 5-node cluster (Nodes A, B, C, D, E) using single-leader replication, Node A is the leader. A hardware failure in a switch causes an asymmetric network partition: Node A can still receive messages from all other nodes, but all outgoing messages from A to B, C, D, and E are dropped.

Scenario 4 of 4

🔀 Trade-Off Tribunal: Synchronous vs. Asynchronous Models

You must make a ruling on the choice of system models and timing assumptions for a new distributed database engine that needs to guarantee linearizable consistency.

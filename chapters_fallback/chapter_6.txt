      "section_title": "1. Single-Leader Replication Architecture",
      "key_concepts": [
        "**Leader-Based Model**: All write requests from clients must be routed to a single designated node (the leader, primary, or active). The leader applies writes to local storage and generates a replication log or change stream.",
        "**Followers**: Non-leader replicas (secondaries, passive standbys, or read replicas) process the replication log to apply writes locally in the exact order determined by the leader. Followers serve read-only queries.",
        "**Synchronous Replication**: The leader blocks and waits for a follower to acknowledge that it has safely recorded the write before returning a success status to the client. This guarantees up-to-date data on the follower but increases write latency and stalls the system if the follower crashes.",
        "**Asynchronous Replication**: The leader returns success to the client immediately after writing to its local storage, sending the replication update to followers asynchronously. This yields highly performant, non-blocking writes but risks data loss if the leader fails before the changes reach the followers.",
        "**Semisynchronous Configuration**: A hybrid approach where one follower is configured to be synchronous and the rest are asynchronous. If the synchronous follower goes offline, an asynchronous follower is promoted to synchronous to ensure that at least two nodes constantly hold a complete copy of the data."
      ],
      "trade_offs_and_mechanisms": [
        "**Setting Up New Followers**: Initiated by taking a consistent snapshot of the leader without locking the database, copying it to the new node, and requesting all replication log changes since the snapshot's log coordinate (e.g., Log Sequence Number or binlog coordinates).",
        "**Catch-up Recovery for Followers**: If a follower crashes, it utilizes its local log to identify the last processed transaction and requests only the subsequent modifications from the leader upon reboot.",
        "**Automatic Failover Steps**: 1) Detection via a heartbeat timeout (e.g., 30 seconds of inactivity). 2) Election or appointment of a new leader (often the follower with the highest LSN). 3) Reconfiguration of client routing and instructing the old leader to become a follower upon its return.",
        "**Failover Vulnerabilities**: Discarding un-replicated asynchronous writes can lead to data loss and coordinate failures across external systems (e.g., primary key reuse causing divergence between a relational database and a cache). Unnecessary failovers can also be triggered by temporary load spikes or network latency."

      "key_concepts": [
        "**Statement-Based Replication**: The leader logs and forwards every raw write query (e.g., SQL statements) to its followers. Extremely compact but highly vulnerable to nondeterministic operations (e.g., `NOW()`, `RAND()`, autoincrements, or state-dependent triggers).",

      "section_title": "4. Managing Replication Lag Anomalies",
      "key_concepts": [
        "**Read-Your-Writes Consistency**: Prevents users from submitting an update and subsequently loading a stale replica that lacks their write. It is guaranteed by reading highly dynamic data (e.g., the user's own profile) exclusively from the leader, or waiting to route queries to a follower until its LSN catches up to the client's write timestamp.",
        "**Monotonic Reads**: Guarantees that an individual user will not observe the database state roll backward in time during successive reads.         "**Consistent Prefix Reads**: Ensures that causally related writes are observed in their correct logical order (preventing an observer from seeing an answer before a question). This is a critical issue in sharded databases because separate partitions operate independently with no global write order. Solutions involve routing causally linked writes to the same partition or explicitly tracking causal dependencies."
